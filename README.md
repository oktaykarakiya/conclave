# Conclave

A portable, web-driven **autonomous AI coding team** you can attach to any git repository.

Conclave drives **opencode** (headless) through a proven loop —
**plan → develop → multi-agent review → grounded verdict → green-gate → commit & merge** —
fully autonomously, controlled entirely from a web UI (you never edit files in the repo, not
even config).

> **Engine migration in progress.** Conclave is moving to an **opencode-native** architecture:
> the engine, model/provider selection, repo context, and usage reporting all defer to opencode.
> Select it at boot with `CONCLAVE_ENGINE=opencode`. The legacy `claude`-CLI engine remains as a
> transitional fallback.

## Highlights

- **Attach to any repo** — runs as a host daemon and operates via isolated git *worktrees*, so your
  working tree is never touched and the repo is never polluted.
- **Agent-ception planning** — a dedicated page where AI agents discuss and decompose a big goal into
  a reviewed, collapsible task tree; you can interject, and on approval the tasks land in the queue.
- **Web UI is the only control surface** — projects, tasks, live logs, and quarantine, all in the
  browser. Config lives in SQLite, not YAML files. Single-page-per-tab, responsive (works from your
  phone).
- **opencode-native engine** — agents run on **opencode**, so model and provider selection lives in
  opencode itself. Authenticate a provider (`opencode auth login`) and set a default model in
  `~/.config/opencode/opencode.jsonc` — Conclave inherits whatever model opencode is configured to
  use. **DeepSeek** is the intended default.
- **Repo intelligence** — repo context comes from an `AGENTS.md` generated by opencode's `/init`,
  which opencode reads natively when agents run — no separate onboarding/analysis step.
- **Grounded verification + green-gate** — reviewer/security/tester verdicts must cite evidence that
  exists in the actual diff (hallucinated findings are downgraded), and `ruff + mypy + pytest` must be
  green before anything merges. Per-worktree venv provisioning so reviewers and the gate share the
  toolchain.
- **Hardened engine** — atomic SQLite transactions, dependency-ordered task claims, crash-safe task
  lifecycle, reviewer-retry with a "no merge without a real review" guard, and bounded resource use.

## Prerequisites

- **opencode** installed and on your `PATH` (via npm — `npm install -g opencode-ai` — or the official
  installer). Conclave drives it as the engine. Authenticate a provider and pick a model in opencode
  itself (see Quick start). The `claude` CLI is **no longer required** when running on the opencode
  engine.
- **Python ≥ 3.12** and **git**. (Node ≥ 18 is only needed for frontend development.)

## Quick start

First, configure opencode (one time) — this is where the model and provider live:

```bash
opencode auth login             # authenticate a provider (e.g. DeepSeek)
# then set a default model in ~/.config/opencode/opencode.jsonc, e.g.:
#   { "model": "deepseek/deepseek-v4-pro" }
opencode run "say hi"           # optional: smoke-test headless before wiring up Conclave
```

Then start Conclave on the opencode engine:

```bash
CONCLAVE_ENGINE=opencode ./conclave   # bootstraps a venv on first run, then starts the daemon
# open http://127.0.0.1:8700  (or http://<your-LAN-ip>:8700 from another device)
```

Conclave inherits whatever model opencode is configured to use — there's no per-run model flag and
no in-app model/secrets config. To change models or providers, reconfigure opencode.

Then, entirely in the web UI:

1. **Attach project** — give it a name and the absolute path to a git repo. Repo context is supplied
   by an `AGENTS.md` (generate it with opencode's `/init`), which opencode reads natively when agents
   run.
2. **Plan or create a task** — use *Agent-ception* to decompose a big goal, or create a task directly;
   approve it (or auto-approve) and watch the team on the **Live** tab: develop → grounded review →
   green-gate → commit → merge into your target branch.

Conclave never touches your working tree: each task runs in an isolated git *worktree* under
`~/.local/share/conclave/` (override with `CONCLAVE_HOME`). Host/port via `CONCLAVE_HOST` /
`CONCLAVE_PORT`.

## Security model

Conclave is a **single-user, personal tool** and is **intentionally unauthenticated** — it can
configure and run shell commands on the host (that's the point). It defaults to binding `0.0.0.0`
so you can reach the UI from your phone on your home network. **Run it only on a trusted network.**
Set `CONCLAVE_HOST=127.0.0.1` to restrict it to loopback. Multi-user auth/RBAC is out of scope.

## Status

The autonomous task-queue engine, Agent-ception planning, web UI, and quarantine are complete and
hardened (`ruff` + `mypy --strict` clean). The **opencode-native** engine migration is in progress:
agents run on opencode, with model/provider selection, repo context (`AGENTS.md` via `/init`), and
cost/token usage all deferred to opencode (usage is read from opencode's reported `step_finish`
events). The legacy `claude`-CLI engine stays available as a transitional fallback. On the roadmap
(see `TODO.md`): the continuous **Autonomous Bug-Fixer** mode, BMad-style scale-adaptive planning,
post-mortem agent, notifications, and true token streaming.

## Development

```bash
python -m venv .venv && . .venv/bin/activate
pip install -e ".[dev]"
pytest -q                    # deterministic; a fake provider stands in for the engine — no LLM cost
ruff check src tests && mypy # lint + strict types

cd frontend && npm install && npm run build   # rebuild the SPA into the daemon's static dir
npx tsc --noEmit                              # strict TS
```
